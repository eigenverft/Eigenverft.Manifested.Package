# Team catalog trust

Notes on how team package JSON should be trusted, signed, and shown to the user. This backs up the **Team catalog trust** items in [PROJECT-TODO.md](PROJECT-TODO.md).

**This document is the design record.** The v1 implementation now exists in the module; notes below call out what is implemented and what remains future work.

## Requirements now chosen

Today, anyone who can edit a file on a team share can change install URLs, versions, or behavior. The JSON can still say `"publisherId": "My Team"` - that string alone proves nothing.

The target trust design is now:

- **Schema-bound trust, not endpoint trust.** Endpoints stay scan locations only. Catalog trust belongs to the package definition plus local publisher/key policy.
- **Embedded definition signature.** The signature lives inside the package-definition JSON so one file can be copied, reviewed, generated by tools, and signed without sidecar coordination.
- **Whole definition is signed.** Verification covers the full semantic definition, including install URLs, hashes, display text, dependencies, and operations. The embedded signature value itself is excluded from the signed payload.
- **Canonical JSON, not source formatting.** Verification must not depend on hand-written property order or whitespace. The signer and verifier need one deterministic canonicalization rule for the definition object minus the signature value.
- **Signing is isolated in helper functions.** The helper loads JSON, builds the canonical signable object, signs/verifies bytes, injects/removes only the signature value, and writes valid JSON. Users and agents should not hand-edit signature bytes.
- **Compatibility-first crypto.** Use RSA with SHA-256 and an X.509 certificate/public key record for v1. This works with Windows PowerShell 5.1/.NET Framework better than newer key types such as Ed25519.
- **No certificate chains in v1.** The first version trusts pinned keys/thumbprints, like SSH/RDP trust-on-first-use. Chain/PKI support can be added later behind the same verification boundary.
- **Trust is by key thumbprint.** The prompt and status display show publisher id/name and certificate/key display fields, but the durable trust decision is the key thumbprint.
- **Trust inventory is the authority.** `PackageTrustInventory.json` is the catalog authority for trusted keys, scoped by `publisherId`; endpoint rows only discover definitions and publisher ids are labels inside signed JSON.
- **Global catalog trust policy.** Add a `PackageConfig.json` policy such as `strict` or `allowUnsigned`. This is not endpoint trust and should not live on endpoint rows.
- **Invalid signature always fails.** A bad signature is not a warning state. It means the definition was tampered with or signed incorrectly.
- **Unknown signed key is not trusted automatically in v1.** Because the embedded signature does not carry `certificatePem`, runtime cannot verify an unknown key from the definition alone. Unknown keys fail and tell the operator to preseed trust with `Trust-PackageSigningCertificate` or `Import-PackageTrust`. An interactive trust prompt can be added later if we add a certificate-distribution path.
- **Unsigned definitions are explicit migration/dev behavior.** `allowUnsigned` only permits unsigned definitions when `publisherId` is listed in `catalogTrust.allowUnsignedPublisherIds`. `strict` rejects all unsigned definitions.
- **Shipped Eigenverft definitions are signed.** The module ships signed definitions and verification keys, then seeds those keys into local trust on first use.
- **Changed key means changed trust.** If a publisher changes key/thumbprint, the local store does not trust the new key until it is shipped by the module or explicitly trusted by the user.
- **Revocation is exact-thumbprint and local first.** v1 should be able to revoke or block a signing key by exact thumbprint before any wildcard, online revocation list, or certificate-chain validation work.
- **Payload strictness is separate.** Catalog signing trusts the definition author. Payload hashes/signatures protect downloaded or depot files. The selected default is `enforceWhenPackageFileExists`, implemented behind a policy helper so it can be changed later without rewriting signature verification.
- **Always print trust status before planning.** `Invoke-Package` should show publisher, endpoint, revision, signature state, key/thumbprint, and trust source before package selection or install work continues.
- **Authoring stays tool-driven.** A human/agent can generate JSON, then a maintainer runs a signing command. CI may validate signatures and schema, but it is not required to be the signer.
- **Use a new breaking schema for shipped/current definitions.** Catalog signing belongs in schema 1.7. Current shipped definitions move to 1.7; runtime keeps limited 1.6 compatibility for old local/test definitions, but 1.6 is not the catalog trust target.

## Source-checked fit

These are the codebase constraints that shaped the decisions above:

| Source area | What it means for trust design |
| --- | --- |
| `PackageEndpointInventory.json` and `Assert-PackageEndpointSource` | Endpoint rows reject retired trust fields (`trusted`, `trustMode`, `trustedAtUtc`, `trustReason`). Adding endpoint trust back would conflict with the current model. |
| `definitionPublication` in schema 1.6 | The current durable identity is already `publisherId` + `definitionId` + `definitionRevision`, so catalog signature metadata should sit with publication metadata. |
| `DefinitionSchema.ps1` | Runtime supports schema `1.6` and `1.7`. Schema `1.7` adds `definitionPublication.definitionSignature`; shipped definitions now use it. |
| `eigenverft-module-package-definition-1.6.schema.json` | The schema uses `additionalProperties: false`; adding signature fields to 1.6 would be a real schema change anyway. |
| `PackageTrustInventory.json` | Catalog authority is cryptographic and key-based: trusted key rows are scoped to `publisherId`, and revoked/blocked key rows fail before package planning. |
| `Resolve-PackageDefinitionReference` | Candidate selection now assesses signature/trust before winner selection. Signed trusted definitions do not require a publisher inventory row. Unsigned definitions require `allowUnsigned` plus `catalogTrust.allowUnsignedPublisherIds`. |
| `Copy-PackageDefinitionToLocalDefinitionStore` | Local candidate and assigned snapshots already store the selected definition hash. Trust status can be recorded near this path without changing depot behavior. |
| Definition resolution logging | Resolution prints a `[TRUST]` line with signature status, catalog trust status, key/thumbprint, signer, and policy before package selection/install work continues. |
| `LocalEnvironment.ps1` and `Bootstrap.ps1` | First-use local inventory copying exists for config, endpoint, depot, and trust inventories. Publisher inventory is no longer copied or shipped. |
| Artifact `contentHash` / `publisherSignature` | Payload trust belongs to artifacts. Catalog signing proves who authored the definition; artifact hashes/signatures prove downloaded payloads. `catalogTrust.payloadVerification` can require that boundary when a package file exists. |

## What the product does today

### Three pieces

```text
PackageEndpointInventory.json   ->  where to look for JSON (module folder, team share, ...)
PackageTrustInventory.json      ->  which signing keys/thumbprints are trusted or blocked
PackageConfig.catalogTrust      ->  strict vs unsigned migration policy
Package definition JSON (1.7)   ->  what to install, plus embedded definition signature
```

You **cannot** put trust on an endpoint row. Endpoints discover JSON only. `PackageTrustInventory.json` decides cryptographic trust by exact key thumbprint scoped to `publisherId`; `PackageConfig.catalogTrust` decides whether unsigned migration is allowed for specific publisher ids.

Before install/planning, the engine assesses the definition signature and then picks a winner:

- `strict` accepts only signed definitions whose key is trusted for the publisher.
- `allowUnsigned` accepts trusted signatures, or unsigned definitions only when `publisherId` is listed in `catalogTrust.allowUnsignedPublisherIds`.
- A signed definition with an invalid, revoked, unknown, or untrusted key fails instead of falling back to unsigned trust.

### First run on a PC

The first time you run `Invoke-Package` on a profile:

1. Folders are created under `%LOCALAPPDATA%/Programs/Evf.Package`.
2. Local copies are made from the module if missing:
   - `PackageConfig.json`
   - `PackageEndpointInventory.json`
   - `PackageDepotInventory.json`
   - `PackageTrustInventory.json`

For shipped packages you do not need manual setup: the shipped Eigenverft signing certificate is seeded into local trust. Team publishers are not auto-trusted; use trust/import commands for signing certificates, or `catalogTrust.allowUnsignedPublisherIds` only for explicit unsigned migration.

### Existing `signatures` fields are not catalog signing

In schema 1.6, `signatures` under discovery/readiness means Authenticode checks on installed files. `packageFile.publisherSignature` means a payload/executable signature. Neither is a signature on the package-definition JSON.

Catalog signing needs a new publication-level field, not reuse of those payload/readiness fields.

### Other protections today

- Same `definitionRevision` with different file content across endpoints fails.
- Many packages already pin payload hashes, but hash presence is not globally required.
- Strict catalog mode is the shipped default in `PackageConfig.json`.

### Payload verification today

The package-file path already has usable trust hooks:

- `targetArtifacts.<target>.contentHash` supports `sha256` and `sha512`.
- `targetArtifacts.<target>.publisherSignature` supports Authenticode checks for executable payloads.
- Each `acquisitionCandidate` has `verification.mode = required | optional | none`.
- Runtime combines acquisition-candidate verification with the selected package file's `contentHash` / `publisherSignature`.
- A staged/depot file is accepted only when its effective verification accepts it.

With `catalogTrust.payloadVerification = enforceWhenPackageFileExists`, package-file acquisition requires `packageFile.contentHash` or `packageFile.publisherSignature` and forces effective verification mode to `required`. Configs without `catalogTrust` now default to `strict`; old unsigned configs must opt into `allowUnsigned`.

## Target schema shape

Names may still change, but the required shape should be close to this:

```text
package definition JSON (schemaVersion 1.7)
  definitionPublication
    publisherId
    publisherName
    definitionId
    definitionRevision
    publishedAtUtc
    definitionSignature
      kind                    -> signed | unsigned
      format                  -> embedded-json-rsa-sha256-v1
      signedContent           -> canonicalDefinitionExcludingSignatureValue
      keyThumbprint           -> stable key/cert thumbprint
      signerDisplayName       -> short display text
      certificateSubject      -> optional display/audit text
      signedAtUtc             -> audit time
      signatureValue          -> base64 signature bytes, only when kind=signed
```

The `definitionSignature` field is required in schema 1.7. For signed definitions, `signatureValue` is excluded from the signed content. For unsigned definitions, `kind = unsigned` is still explicit and policy-controlled.

`definitionIntegrity` is not required as a separate trust primitive if the signature covers the canonical definition. A separately displayed digest can still be useful for diagnostics, but the signature is the authority.

## Canonical signing helper

Signing and verification need to use one shared helper path. This should be implemented before wiring trust into package selection, because the signing operation itself is the fragile part.

Signing flow:

1. Load the definition JSON through a signing-aware helper.
2. Ensure `definitionPublication.definitionSignature` exists with all non-secret metadata:
   - `kind = signed`
   - `format = embedded-json-rsa-sha256-v1`
   - `keyThumbprint`
   - signer/certificate display fields
   - `signedAtUtc`
3. Remove or set empty only `definitionPublication.definitionSignature.signatureValue`.
4. Canonicalize the entire definition object.
5. Sign the canonical UTF-8 bytes with RSA/SHA-256.
6. Store the signature bytes as base64 in `signatureValue`.
7. Write the complete JSON file back through the helper.

Verification flow:

1. Load the definition JSON.
2. Read and base64-decode `signatureValue`.
3. Clone the object and remove only `definitionPublication.definitionSignature.signatureValue`.
4. Canonicalize with the same helper used for signing.
5. Verify the canonical UTF-8 bytes against the public key/certificate selected by `keyThumbprint`.

The helper should own the tricky parts:

- preserve JSON validity when injecting the base64 signature
- avoid manual string splicing in normal command code
- make source property order irrelevant to trust
- produce stable output so signing the same semantic object is reproducible apart from fields intentionally changed before signing, such as `signedAtUtc`
- make verification a reverse operation of signing, not a second hand-built parser

Recommended implementation shape:

```text
Get-PackageDefinitionSignableContent
  input: definition object or path
  output: canonical UTF-8 bytes, signature metadata, original signature value

Set-PackageDefinitionSignatureValue
  input: path/object + base64 signature value
  output: valid signed JSON

Test-PackageDefinitionSignatureValue
  input: path/object + public key/certificate
  output: signature status + canonical content hash + signer/key metadata
```

The exact canonicalization contract is still a coding detail, but it should be hidden behind these helpers. If we later improve canonicalization, command behavior should change in one place.

## Target local trust store and options

Use a new inventory, not the Windows certificate store for v1:

```text
PackageTrustInventory.json
  inventoryVersion
  keys[]
    publisherId
    publisherName
    keyThumbprint
    certificatePem
    signerDisplayName
    trustSource              -> moduleShipped | userApproved | imported | ciPreseed
    enabled                  -> true/false
    trustedAtUtc
    trustReason
    revokedAtUtc             -> optional local revocation marker
    revocationReason         -> optional local note
  revokedKeys[]              -> optional exact-thumbprint block list
```

Recommended default path:

```text
%LOCALAPPDATA%/Programs/Evf.Package/Configuration/Internal/PackageTrustInventory.json
```

`PackageConfig.json` should carry global behavior policy and may carry a path override later:

```json
{
  "package": {
    "catalogTrust": {
      "policy": "strict",
      "trustInventoryFilePath": "{applicationRootDirectory}/Configuration/Internal/PackageTrustInventory.json"
    }
  }
}
```

If the first implementation wants to reduce bootstrap risk, it can hard-code the same default helper first and add the `PackageConfig.json` override in the next step. The important requirement is to isolate the path behind helper functions so the storage can move later without touching verification logic.

Do not keep a separate publisher inventory in the runtime trust path. Publisher display data belongs in signed `definitionPublication` and trust summaries; coarse policy belongs in `PackageConfig.catalogTrust`; key material belongs in `PackageTrustInventory.json`.

Storage options considered:

| Option | Fit | Tradeoff |
| --- | --- | --- |
| **New `PackageTrustInventory.json`** | Implemented v1 path. Easy to seed, inspect, back up, and scope by publisher id. | Adds one dedicated inventory file and helper set. |
| **Extend `PackagePublisherInventory.json`** | Rejected for the runtime path. | Mixes name policy, unsigned escape policy, and key material in one file; gets awkward for key rotation and multiple keys per publisher. |
| **Directory of one trust file per key** | Friendly for manual copy/sync and easier conflict handling if teams distribute individual trust records. | Requires scanning and merge logic; more filesystem edge cases; more files to explain. |
| **Windows certificate store** | Natural for enterprise PKI and cert UI. | Harder for local/portable/team-share workflows, harder to test, and less transparent for this package engine's own inventory story. |

Recommended path: ship `PackageTrustInventory.json` and keep publisher inventory out of normal runtime resolution. Helpers should still allow a later directory-backed trust store without changing verification.

## Verification flow

Target flow before any package planning/install work:

1. Read the candidate JSON and basic publication metadata.
2. Validate schema version and required publication/signature fields.
3. Build canonical signed content from the whole definition, excluding only `definitionPublication.definitionSignature.signatureValue`.
4. Verify the RSA/SHA-256 signature when `definitionSignature.kind = signed`.
5. Match `keyThumbprint` against enabled, non-revoked records in `PackageTrustInventory.json`.
6. Apply global `PackageConfig.json` catalog trust policy:
   - `strict`: signed, valid, and trusted key required.
   - `allowUnsigned`: signed trusted definitions pass; unsigned definitions pass only when `publisherId` is listed in `catalogTrust.allowUnsignedPublisherIds`.
7. Apply coarse publisher blocks from `catalogTrust.blockedPublisherIds`.
8. Apply existing publisher conflict and revision selection rules.
9. Print trust status before package resolution continues.

Important implementation consequence: candidate rows need a trust assessment before the winner is chosen, otherwise signature status, unknown-key errors, and unsigned migration policy cannot be explained clearly.

## Payload strictness direction

Keep payload strictness out of catalog trust. The selected default is:

- Catalog trust answers: "was this definition written by a trusted key?"
- Payload trust answers: "does this downloaded/depot file match the signed definition's expected hash/signature?"

Target policy:

```json
{
  "package": {
    "catalogTrust": {
      "policy": "strict",
      "payloadVerification": "enforceWhenPackageFileExists"
    }
  }
}
```

Candidate meanings:

| Option | Meaning | Fit |
| --- | --- | --- |
| `off` | Keep current package-defined behavior. | Good migration/default escape. |
| `warnWhenPackageFileExists` | Warn when a saved package-file path lacks `contentHash` or `publisherSignature`. | Good discovery/reporting step. |
| `enforceWhenPackageFileExists` | If the selected install path requires a saved package file, require `contentHash` or `publisherSignature` and require verification. | Best strict-mode target for archive/MSI/installer/download/depot files. |
| `enforceAllAcquisition` | Require every acquisition candidate to have enforced verification. | Too broad for v1 because not every install kind uses the same package-file model. |

This means a manipulated depot is caught when:

1. the signed definition contains the expected payload hash/signature, and
2. the acquisition path verifies the depot file before reuse.

If either is missing, catalog signing still proves who authored the JSON, but it does not prove the depot file was not replaced.

## Revocation direction

Start with local revocation/blocking, not online revocation infrastructure.

Required v1 shape inside `PackageTrustInventory.json`:

```text
PackageTrustInventory.json
  keys[]
    keyThumbprint
    publisherId
    enabled
    revokedAtUtc
    revocationReason

  revokedKeys[]
    keyThumbprint          -> exact match, preferred
    publisherId            -> optional exact publisher scope
    reason
    addedAtUtc
```

Options:

| Option | Meaning | Tradeoff |
| --- | --- | --- |
| Exact key disable | Set `enabled = false` or `revokedAtUtc` on a trusted key record. | Simple and safe; requires the key to be known locally. |
| Exact revoked key list | Keep `revokedKeys[]` even if the full key record is missing. | Good for module-shipped emergency blocks and imported block records. |
| Publisher block | Use `catalogTrust.blockedPublisherIds[]` to block a whole exact publisher id locally. | Coarse but useful for emergency policy; no wildcard support in this pass. |
| Wildcard publisher block | Block all keys for `publisherId` or a pattern. | Later only. Exact ids are safer and clearer for v1. |
| Wildcard thumbprint block | Pattern-match thumbprints. | Avoid for v1; exact thumbprints are safer and clearer. |
| Certificate expiry / chain / CRL / OCSP | Treat X.509 like enterprise PKI. | Later only. Adds network and chain semantics that are not needed for lightweight self-signed trust. |

Default behavior: exact thumbprint trust and exact thumbprint revocation first. Allow publisher-level blocking through `catalogTrust.blockedPublisherIds[]`. Keep `revokedKeys[]` in `PackageTrustInventory.json` for v1; a separate revocation inventory is a later option only.

## User experience

Each `Invoke-Package` run should print one trust line early, for example:

```text
[TRUST] definition='VSCodeRuntime' publisher='Eigenverft' endpoint='moduleDefaults' revision='2' signature='valid' key='ABCD1234...' trust='moduleShipped'
```

Status names should be small and stable:

| Status | Meaning | Default action |
| --- | --- | --- |
| `validTrusted` | Signature verifies and key is trusted | Continue |
| `validUnknownKey` | Signature verifies but key is not trusted | Prompt if interactive; fail if noninteractive |
| `unsignedAllowed` | No signature, but `allowUnsignedPublisherIds` allows this publisher id | Warn and continue |
| `unsignedBlocked` | No signature and policy does not allow it | Fail |
| `invalidSignature` | Signature exists but does not verify | Fail |
| `revokedKey` | Key exists but is disabled/revoked locally | Fail |

Interactive behavior:

- Unknown signed key may prompt once per key thumbprint.
- The prompt must show publisher id/name, definition id/revision, signer display name, certificate subject if present, key thumbprint, endpoint, and source path.
- Trusting the key stores the key/thumbprint in `PackageTrustInventory.json`.

Noninteractive behavior:

- Never prompt.
- Unknown signed key fails with a command hint to import/preseed trust.
- Unsigned definitions only run when `catalogTrust.policy = allowUnsigned` and `publisherId` is listed in `catalogTrust.allowUnsignedPublisherIds`.

## Team and AI workflow

Base workflow:

1. A human or LLM edits package-definition JSON.
2. Validation checks schema, references, payload hashes, and product-boundary rules.
3. A maintainer signs the JSON with a local or team-controlled key.
4. The signed JSON is copied to the endpoint.
5. User machines verify and trust by key thumbprint.

For Eigenverft shipped definitions:

- The definitions are signed before the module is packaged.
- The module ships the public verification key/certificate.
- First local setup seeds Eigenverft trust into `PackageTrustInventory.json`.

For CI:

- CI can validate signed definitions.
- CI can sign if a team wants that, but it is not the required base model.
- Automation should preseed/import trust before invoking package assignment; it should not rely on an interactive trust prompt.

Signing tools should make the common path small:

- create or import a self-signed/catalog-signing certificate
- sign one package-definition JSON
- verify one package-definition JSON
- verify a folder of definitions
- export/import trust records for noninteractive or team setup

Private-key/password UX is the ugly part and should be handled by tooling, not by asking users to hand-edit signature fields.

## Locked implementation requirements

There are no remaining open design questions in this section. The items below are the decisions to implement first; alternatives are recorded only as rejected or later options so the coding work has a stable target.

### 1. Canonicalization algorithm

**Requirement: use an internal canonical JSON helper.**

Sign the full package-definition object, excluding only `definitionPublication.definitionSignature.signatureValue`. The signature value is injected after signing. Verification removes that same value and runs the same helper in reverse.

Raw file/text signing is not the v1 target. The implementation must parse JSON, produce deterministic canonical bytes, and make normal package code call the helper instead of hand-splicing JSON text.

Reason:

- Signing the signature value itself is impossible because the value does not exist until after signing.
- Signing raw file bytes would make trust depend on whitespace, property order, and editor formatting.
- Signing only selected fields would leave room for important definition content to change without invalidating trust.
- A single helper keeps the fragile "remove signature value, canonicalize, sign/verify, inject value" operation isolated from normal package logic.

Implementation requirement:

- Parse JSON into an object model.
- Clone the object for signing/verification.
- Remove only `definitionPublication.definitionSignature.signatureValue`.
- Sort object properties deterministically.
- Preserve array order.
- Emit minified UTF-8 JSON with stable string escaping and invariant primitive rendering.
- Return canonical bytes plus a diagnostic content hash for tests and troubleshooting.

First proof: implement the canonicalization package helper first, then prove it by signing a backup copy of one existing definition and verifying it. Do not rely on the current object order emitted by `ConvertTo-Json` unless the helper deliberately constructs that order.

### 2. Trust inventory fields and key material

**Requirement: use `PackageTrustInventory.json` with `certificatePem` plus derived metadata.**

Use a local `PackageTrustInventory.json` as the v1 trust store. Store `certificatePem` plus derived metadata. Start with a default path helper and add the `PackageConfig.json` path override later unless implementation proves the override is cheap.

Purpose of `certificatePem`:

- It stores the public X.509 certificate only. It must never contain the private key.
- It carries the RSA public key needed for definition-signature verification.
- It carries useful display and audit fields such as subject, issuer, validity dates, serial number, and thumbprint.
- It fits the self-signed workflow: the maintainer creates a signing certificate once, keeps the private PFX private, and distributes only the public certificate/trust record.
- It gives `Get-PackageTrust` and trust prompts better human text than a raw key blob.
- It leaves room for later chain/expiry checks without making certificate chains a v1 requirement.

Required v1 trust row fields:

- `publisherId`, `publisherName`
- `keyThumbprint`
- `certificatePem`
- `certificateSubject`, `certificateIssuer`, `certificateSerialNumber`
- `notBeforeUtc`, `notAfterUtc`
- `signerDisplayName`
- `trustSource`: `moduleShipped`, `userApproved`, `imported`, `ciPreseed`
- `trustedAtUtc`, `trustedBy`
- `enabled`
- `revokedAtUtc`, `revocationReason`, `revokedBy`

Implementation requirement: the command that imports trust can accept either a certificate file or exported trust record, but the inventory should normalize to `certificatePem` plus derived metadata.

Path requirement: add `Get-PackageTrustInventoryPath` beside the existing inventory helpers and point it to the local internal configuration area. Keep `PackageConfig.json` path override and directory-of-trust-files support as later options.

### 3. Command names

**Requirement: use the clearest user-facing verbs, even though they are not approved PowerShell verbs.**

Use one command family for signing definition files and another for managing trusted public keys/certs. Avoid short combined spellings in exported command names. Prefer `Definition`, `Signature`, and `Certificate` over `Def`, `Sig`, and `Cert`.

PowerShell verb note:

- `Sign`, `Verify`, `Trust`, and `Untrust` are not approved PowerShell verbs.
- We accept that tradeoff because the commands should read like the user task.
- `Revoke`, `Import`, `Export`, `Get`, `New`, and `Remove` are approved and can still be used where they are already clear.

Required command set:

| Command | Purpose |
| --- | --- |
| `New-PackageSigningCertificate` | Create a local self-signed certificate suitable for catalog signing. |
| `Sign-PackageDefinition` | Sign a package-definition JSON and inject/update `signatureValue`. |
| `Verify-PackageDefinitionSignature` | Verify one package-definition JSON without installing. |
| `Verify-PackageDefinitionCatalog` | Verify a folder of definitions without installing. |
| `Get-PackageSigningProfile` | Show local non-secret signing profile metadata. |
| `Get-PackageTrust` | Show trusted/revoked key/cert records. |
| `Trust-PackageSigningCertificate` | Trust a public signing certificate/key thumbprint for a publisher. |
| `Untrust-PackageSigningCertificate` | Remove or disable local trust for a signing certificate/key thumbprint. |
| `Import-PackageTrust` | Import an exported trust record or certificate file. |
| `Export-PackageTrust` | Export a public trust record for CI/team preseed. |
| `Revoke-PackageSigningCertificate` | Mark a trusted key/thumbprint as revoked. |
| `Block-PackageSigningCertificate` | Add an explicit thumbprint block even if there is no approved trust row. |
| `Remove-PackageTrust` | Remove a local trust record. |

### 4. Private-key and signing UX

**Requirement: use a password-protected PFX file with a local DPAPI-protected signing profile for the friendly path.**

Users should not manually type signature fields. The module should provide tooling for self-signed/local signing and verification. The Windows certificate store, Windows Credential Manager, and CI secret signing are later options, not v1 requirements.

Base workflow:

1. `New-PackageSigningCertificate -Name <publisher> -Password <securestring>` creates a self-signed signing certificate, writes a password-protected PFX, writes a public `.cer`, writes a public trust export JSON, and saves a local DPAPI-protected signing profile.
2. The maintainer keeps the PFX private.
3. Clients import the public `.cer` with `Import-PackageTrust`; the trust export JSON remains supported for richer metadata or preseed scenarios.
4. `Sign-PackageDefinition -Path <definition.json>` reads the profile for `definitionPublication.publisherId`, decrypts the password for the current Windows user, signs the canonical definition bytes, and injects `signatureValue`.
5. `Verify-PackageDefinitionSignature` verifies the signed file before it replaces a shipped/team definition.

Required UX:

- how a user creates the first self-signed signing cert
- how the private key is protected in the PFX
- how the PFX password is stored locally with DPAPI and never shipped
- how the public `.cer` or trust record is exported
- how a definition is signed without hand-editing JSON
- how a signed backup/copy is verified before replacing shipped definitions
- private-key material and PFX passwords must never be stored in `PackageTrustInventory.json`

### 5. Payload strictness policy

**Requirement: use `enforceWhenPackageFileExists`, designed as a switchable policy.**

Payload strictness is separate from catalog trust. Catalog signing proves who authored the JSON; payload hashes/signatures prove downloaded or depot files match what the trusted author intended.

Current source reality:

- `contentHash` supports `sha256` and `sha512`.
- `publisherSignature` supports Authenticode checks.
- `verification.mode` controls whether missing verification can pass.
- The engine can enforce payload verification when the selected install kind has a saved `packageFile`.
- Not every install kind has the same package-file boundary, so a single global "all downloads must have hashes" rule would be too blunt today.

Policy behavior:

- If the selected package path uses a saved package file, require `contentHash` or `publisherSignature`.
- Require effective verification before reuse/download/depot promotion.
- Do not require every acquisition candidate in every install kind to have the same file-boundary metadata.
- Keep warn-only and stricter all-acquisition modes as later policy values, not v1 defaults.

Implementation requirement: do not hard-code this into the catalog-signature verifier. Put the check behind one policy function, for example `Resolve-PackagePayloadVerificationRequirement`, so we can later change the rule from `enforceWhenPackageFileExists` to warn-only, stricter all-acquisition, or package-kind-specific behavior without rewriting trust verification. A definition can be signed and trusted while still failing payload policy.

### 6. Revocation fields

**Requirement: store revocation in `PackageTrustInventory.json`.**

Purpose of revocation:

- A key/certificate was trusted before, but should no longer verify new definitions.
- A team lost a private key, replaced a signer, or found out a key was copied.
- A module update wants to ship a known-bad thumbprint block.
- A user wants to block a local key without deleting the audit trail of when it was trusted.

Revocation is used by the signature/trust resolver, not by endpoint trust and not by payload hash verification. It answers: "This definition is signed by key X; is key X locally or module-shipped blocked before we even consider accepting it?"

Revocation cannot live inside the signed definition because a compromised key could sign its own "not revoked" metadata. It must live in local/module trust policy checked before a definition is accepted.

Where revocation is checked:

1. Read the definition signature thumbprint.
2. Check local/module revocation policy for that exact thumbprint.
3. If blocked/revoked, fail before trust prompt and before candidate selection.
4. If not revoked, continue normal trusted-key lookup or unknown-key prompt.
5. A later trust import must not silently override a local or module-shipped block for the same thumbprint.

Required storage shape:

```text
PackageTrustInventory.json
  keys[]
    keyThumbprint
    enabled
    revokedAtUtc
    revocationReason
    revokedBy

  revokedKeys[]
    keyThumbprint          -> exact thumbprint
    publisherId            -> optional exact publisher scope
    source                 -> moduleShipped | user | imported
    reason
    addedAtUtc
```

Key-row revocation handles the normal case: "I trusted this key, then revoked it." `revokedKeys[]` handles the block-list case: "never accept this exact thumbprint, even if there is no local trust row." A separate `PackageTrustRevocationInventory.json`, wildcard thumbprint blocks, remote revocation, CRL, OCSP, and chain validation are later options.
